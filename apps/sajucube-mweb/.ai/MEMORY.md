# 기억

## 제품 방향
- 첫 기능은 `만세력 표시`다.
- 설명 카드형 UI보다 실제 만세력 보드가 잘 읽히는 compact UI를 우선한다.
- 모바일에서도 답답하지 않게 보이는 구성을 중요하게 본다.
- 1차 목표 이후 다음 확장 우선순위는 `차샘만세력`이다.

## 구현 메모
- 계산 라이브러리는 `@gracefullight/saju`, 날짜 처리는 `luxon` 조합을 사용한다.
- 테스트는 기능 작업과 함께 유지한다.
- 공유 URL은 현재 폼의 최신 입력값 기준으로 생성한다.
- PWA 서비스 워커는 프로덕션에서만 등록된다.
- 사용자에게 보이는 브랜드 이름은 `SajuCube`를 사용한다.
- 앱 버전의 기준값은 `package.json`의 `version`으로 관리하고, 화면에는 기본적으로 숨기되 `앱 정보`에서만 노출하는 방향을 사용한다.
- 배포 직전에는 `package.json`의 patch 버전을 한 단계 올리고, minor 버전 증가는 사용자가 요청할 때만 진행한다.
- PWA 아이콘 갱신은 빌드별 버전 쿼리와 서비스 워커 캐시 우회 전제가 있어야 반영이 잘 된다.
- 차샘만세력은 일반 만세력 1개를 확장한 화면이 아니라 `부허 본원`, `부허 차력`, `본원`, `차력`, `허자 본원`, `허자 차력` 6개 만세력을 함께 보여주는 구조로 이해한다.
- 일반 만세력의 기본 만세력은 차샘만세력 기준으로는 `본원`에 해당한다.
- 차력은 본원의 양력 생년월일 숫자를 같은 해의 음력 생년월일 숫자로 치환한 만세력으로 보고, 시간이 같을 때 없는 음력 날짜 보정은 앞/뒤 후보의 양력 실제 거리 비교를 우선한다.
- 부허자 계열은 허자 계열과 반대 방향으로, 현재 양력 날짜를 실제 음력 날짜로 바꾼 뒤 그 음력 숫자를 다시 양력 날짜로 놓는 방식으로 이해한다.
- 허자 본원은 차력의 음력 생년월일 숫자를 다시 양력 생년월일 숫자로 치환해 차력과 같은 규칙으로 계산한 결과로 이해한다.
- 허자 차력은 허자 본원의 양력 생년월일 숫자를 다시 음력 생년월일 숫자로 치환한 결과로 이해한다.
- 차샘만세력 화면은 `부허자 2판`, `본원/차력 2판`, `허자 2판`의 세로 그룹 구조를 우선한다.

## 작업 운영 메모
- 의미 있는 기능 변경을 하면 코드 수정만 끝으로 보지 말고 `.ai/PLAN.md`, 필요하면 `.ai/STATUS.md`, `.ai/MEMORY.md`까지 같은 턴에 함께 갱신한다.
- 사용자가 프로덕션 배포를 요청하면 배포 전에 `package.json`과 `package-lock.json`의 patch 버전을 먼저 올린 뒤 배포한다.
- 이 프로젝트에서 작업 완료 기준은 `코드 변경 -> 검증 -> .ai 문서 갱신 -> 배포 요청 시 patch bump 확인 -> 배포` 순서로 본다.

## 입력 규칙
- 생년월일시는 `YYYYMMDD` 또는 `YYYYMMDDHHmm`처럼 붙여 입력할 수 있고, 화면에서는 `YYYY MMDD` 또는 `YYYY MMDD HHmm`로 정리해 보여준다.
- 생년월일시 입력은 타이핑 중간에도 `8자리(날짜)`, `10자리(시까지)`, `12자리(분까지)` 시점에 자동으로 반영되면 좋다.
- 입력 중 자동 반영은 결과 화면은 바로 따라오되, 브라우저 `뒤로 가기`가 중간 입력 상태에 멈추지 않게 히스토리를 과하게 쌓지 않는 편이 좋다.
- 성별과 양력/음력/윤달 선택도 생년월일시 입력과 마찬가지로 바꾸는 즉시 다시 계산되는 편이 좋다.
- 공유 버튼도 사용자가 보기에는 입력 확정 동작처럼 느껴져야 하므로, 최신 폼 값 기준 URL을 만들면서 주소창 상태도 함께 맞춰주는 편이 좋다.
- 모바일에서 키보드를 닫는 동작은 브라우저마다 `blur` 타이밍이 달라 불안정하므로, 키보드 접기에 확정 의미를 기대하기보다 최신 draft 입력과 다른 오래된 자동 반영 응답을 차단하는 쪽이 낫다.
- 프로그램 첫 진입 시 기본 명식은 고정 예시가 아니라 서울 현재 날짜와 시간을 기준으로 보여주는 편이 좋다.
- 시간이 비어 있으면 화면상 시주는 `?`로 표시하고, 계산은 `12:00` 기준 근사값으로 처리한다.
- 절입 시각이나 자시 경계에 가까운 경우 시간 미입력 상태의 결과는 엄밀값과 달라질 수 있다.
- 양력/음력 어느 방식으로 입력해도 표시용 출생일은 양력으로 변환한 값을 기준으로 본다.
- 윤달 입력은 라이브러리가 조용히 일반 음력으로 바꾸는 경우가 있어, 요청한 윤달 여부와 실제 변환 결과가 정확히 일치하는지 한 번 더 검증해야 한다.
- 윤달 입력의 표시 형식은 `윤 YYYY-MM-DD`를 기준으로 삼고, 예시는 `남 홍길동` 다음 줄에 `윤 1972-01-26`처럼 배치한다.
- 사주판 출생 정보 영역에는 양력 생일과 음력 생일을 함께 보여준다.
- 차력의 없는 음력 날짜 보정에서 양력 실제 거리가 같을 때만 `12시 전이면 이전일`, `12시 이후면 다음일` 규칙을 보조적으로 쓴다.

## UI 선호
- 상단 액션은 긴 텍스트 버튼보다 아이콘 버튼을 선호한다.
- `표시 설정` 버튼은 당분간 숨기고, 상단 액션은 생성/공유/앱 정보 정도의 단순한 구성이 낫다.
- 보조 정보(기둥 위치, 적중 위치, 대운/세운 머리말), 대운/세운 구분선, 사주판 배경색은 기본으로 숨기고/끄고, 필요할 때만 설정에서 켜는 방식을 선호한다.
- 공유 이미지는 설명보다 보드 판독성이 더 중요하다.
- 공유 이미지에도 설치 아이콘과 연결되는 작은 브랜드 표식이 있으면 인지가 쉽다.
- 천을귀인/공망 위의 빈 영역에는 `성별 이름`, `양 YYYY-MM-DD HH:mm` 형태의 짧은 출생 정보가 들어가면 좋다.
- 공유 이미지에서는 출생 정보가 천을귀인/공망 바로 위에 붙고, 천을귀인/공망 셀 크기도 주변 타일과 최대한 맞는 구성을 선호한다.
- 차샘만세력 화면에서는 기본 접힘 상태에서 대운/세운을 숨기고, 펼친 판만 하단 운세 영역을 보여주는 방식을 선호한다.
- 차샘만세력 OG는 6개 판의 사주 8자만 2행 3열로 배치하고, 아는 사람 위주 사용을 전제로 설명 라벨은 생략하는 방향을 선호한다.
- 차샘만세력 OG에서는 부허자/허자보다 본원/차력의 판독성이 더 중요하므로, side panel은 일주/월주만 남기고 글자를 더 크게 쓰는 편이 좋다.

## 문서 역할
- `PROJECT.md`: 프로젝트 목표와 범위
- `PLAN.md`: 다음 작업 우선순위
- `STATUS.md`: 현재 구현 상태와 검증 이력
- `DECISIONS.md`: 왜 그렇게 정했는지
## 상호작용 메모
- 대운/세운 선택은 눌렀을 때 즉시 반응하는 느낌이 중요하므로, 가능하면 서버 왕복보다 로컬 선택 갱신을 우선한다.
- 현재 선택 상태는 주소창에도 반영해서 새로고침이나 URL 복사 후에도 보이는 차트 상태와 일치하게 유지한다.
- 차샘만세력은 이제 `부허자 / 본차 / 허자` 상위 그룹 없이 6판을 평평하게 렌더링하고, `본원`과 `차력` 2판만 `isBoncha`로 시각 구분한다.
